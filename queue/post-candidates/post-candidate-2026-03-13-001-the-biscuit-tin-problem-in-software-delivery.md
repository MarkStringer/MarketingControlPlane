# The biscuit tin problem in software delivery

## Hook

A spec is not the software in the same way a biscuit tin is not a biscuit.

## Rationale

This is a strong evergreen candidate because it uses a distinctive manuscript idea—the biscuit tin vs biscuit image—to explain why software delivery is not a matter of copying an idea exactly. It serves the launch goal of tying project management to seeing things differently, without becoming methodology boilerplate.

## Draft

One reason software projects go wrong: people treat the description as if it were the thing.

A roadmap is not the product.
A spec is not the workflow.
A diagram is not the user experience.

The book’s language for this is the move from “exact match” thinking to asymmetrical thinking.

Exact-match thinking says:
- deliver all of it
- exactly as imagined
- with no meaningful change

Reality says:
- users reveal things late
- working software changes the conversation
- value often appears in partial, imperfect form first

That is not failure. That is how discovery works.

Better project management is often just helping people cross this gap:
from the fantasy of an exact match
to something real, useful, and alive.

Deliver some value now. Learn. Then decide what deserves to exist next.

## Why now

Useful at any time because many teams are planning, estimating, or arguing from documents rather than working software. It reinforces a core book argument in plain language.

## Supporting ideas

- Symmetrical thinking creates exact-match fantasy.
- Descriptions and diagrams are not the same as working software.
- Working software helps shift stakeholders toward what is useful now.
- Delivering some value is often smarter than chasing all value.
