# The biscuit tin is not the biscuit

## Hook

A specification can feel finished long before a product is useful. That’s one reason software projects get into trouble.

## Rationale

This is a strong evergreen LinkedIn post built from a distinctive argument in the manuscript rather than generic PM commentary. It uses the biscuit metaphor and exact-match/asymmetry framing to make the idea memorable and practical.

## Draft

One of the most dangerous habits in software delivery is treating the description as if it were the thing.

A diagram is not working software.
A specification is not user value.
The biscuit tin is not the biscuit.

We slip into what the book calls exact-match thinking:
- the idea already exists because we can describe it
- the plan should map neatly onto reality
- any deviation feels like failure

But projects become more deliverable when we think asymmetrically instead:
- some value, not all value
- useful now, not perfect later
- an inexact match, tested in the real world

That’s why showing working software changes the conversation. It breaks the fantasy that words on a page are enough.

In software, the map is never the territory. The only serious question is: what have we learned from the real thing?

## Why now

Useful at any time because it targets a common delivery failure mode: mistaking descriptions, plans, or requirements for value in use.

## Supporting ideas
