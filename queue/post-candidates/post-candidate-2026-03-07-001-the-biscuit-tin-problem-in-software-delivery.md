# The biscuit tin problem in software delivery

## Hook

A spec is not the software in the same way a picture of a biscuit tin is not the biscuit.

## Rationale

This exists to express a core book argument in a vivid, practical way for delivery leaders. It uses the distinctive biscuit-tin framing and the all/same/forever/exact-match lens rather than generic agile advice.

## Draft

One reason software projects go sideways: people treat the description as if it were the thing.

A roadmap, spec, diagram, or backlog can be useful.
But it is not an exact match for working software.

That gap matters.

If stakeholders think the product already exists the moment it is described, every compromise feels like failure:
- prioritising feels like loss
- learning feels like delay
- iteration feels like incompetence
- change feels like betrayal

A better move is asymmetrical thinking.

Instead of asking for all of it, exactly as imagined, forever, ask:
- what is useful now?
- for whom?
- what can we test in the real world?
- what are we learning from the imperfect version?

Projects become more deliverable when we stop chasing exact-match fantasy and start delivering something real, useful, and open to learning.

## Why now

Evergreen idea suited to ongoing awareness-building because it captures a central thesis of the book and ties directly to project management and software delivery.

## Supporting ideas

- Symmetrical thinking drives expectations of perfect matches between idea and product.
- Descriptions and specifications are not exact matches for working software.
- Working software is the best way to shift thinking toward what is useful now.
- Successful delivery depends on asymmetrical thinking: some, different, temporary, inexact.
