This is direct empirical evidence for the distinction the book draws between the idea value stream and the product value stream. Mark can name the pattern that the data is showing: producing more code faster is work in the first stream; getting it to users is work in the second. # Draft reply

The report's finding has a shape: more code, same delivery problems.

There are two different streams at work. One is the stream that generates the idea — the excitement, the feature list, the sprint tickets, the commits. The other is the stream that delivers the product — the thing that actually ships, survives contact with users, and ends up doing something useful.

AI has accelerated the first stream. Apparently by 59%.

But the bottleneck in the second stream isn't about code volume. It's about feedback loops. About showing working software to people who already believe they know what they're getting. About stakeholder ecosystems, political risk, and the gap between what was imagined and what turns out to be useful.

Producing more code faster doesn't close that gap. If anything, more code in the pipeline means more to integrate, more to validate, more chances for the distance between the idea and the product to widen.

The delivery gap is a human and organisational problem. It was before AI. It still is.
