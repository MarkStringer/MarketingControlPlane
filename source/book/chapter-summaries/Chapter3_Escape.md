1. The chapter argues that projects become real when teams get **working software** into the hands of actual users as early as possible.

2. Early working software changes the project from **push** mode, where sponsors drive decisions, to **pull** mode, where user demand starts shaping what happens next.

3. Showing software to real users helps teams discover both **value** and **resistance**: who wants it, who blocks it, and what stands in the way of release.

4. The closer software gets to a real environment, the more it reveals **hidden obstacles** such as accessibility issues, security demands, deployment problems, and organizational blockers.

5. These surprises are useful because they expose the project’s **unknown unknowns** before too much time and money are spent.

6. The chapter defends the Agile focus on **working software over comprehensive documentation**, arguing that software projects need learning and adaptation more than fixed upfront plans.

7. Users react to working software very differently from how they react to specifications, because software creates an immediate, emotional sense of what is useful, frustrating, or exciting.

8. Releasing early will often annoy sponsors, designers, developers, testers, and compliance people, but it is still the safest way to learn whether the software is genuinely wanted.

9. Progress should happen in at least four dimensions at once: **functionality, data, users, and environment**, rather than only adding more features.

10. The chapter’s main message is that what feels risky—promoting software toward real use as soon as possible—is usually the **safest path to success**, because it creates learning, momentum, and user pull.
